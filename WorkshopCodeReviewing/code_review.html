Code review in a nutshell

You get a paper to review. The authors developed a software upon which the results depend. How do you review the code?

  • The main goal of a code review is to ensure that the software works as expected and reproduces the results of the paper without bugs
  • Secondary goals include checking whether the code is clear, maintainable, testable
  • In no way the goal is to create a perfect copy of your personal taste
  • This is not the time to say “I would have done this in a different way”

Software as data

FAIR principles apply to research software as well!

  • F: “Software, and its associated metadata, is easy for both humans and machines to find”
  • A: “Software, and it’s metadata, is retrieavable via standardised protocols”
  • I: “Software interoperates with other software by exchanging data and/or metadata, and/or through interaction via application programming interfaces (APIs), described through standards”
  • R: “Software is both usable (can be executed) and reusable (can be understdood, modified, built upon, or incorporated into other software)”

How to facilitate code review?

Several aspects make your job as a reviewer easier:

  • The tool used to write the software:
    • NetLogo needs a software to be downloaded but it’s straightforward
    • R is relatively easy as many of us have it on the computer
    • Python can be easy, but it depends on what library one has to download
  • Documentation:
    • is there at least a README explaining what the software is, and how it can be used?
  • Folder structure
    • is there only one folder with a mess of files? Are main files well explained?
  • Is there data involved? Did the author make available a way to obtain necessary data?

Code review: the author’s perspective

As software developer, you have to make your code easier to review! The code needs to be:

  • readable (also after 6 months)
  • modifiable (by someone else, if necessary)
  • testable (and tested!)

These things do not only make the code reviewable, but they make it maintainable.

Programming to make code readable

You can use code as documentation of your processes.

Compare this:

a<-seq(0,100,0.1)
b<-dnorm(a,40,2)
plot(a,b,type="l",main="Normal distribution",lty=2)

To this:

# generate normal distribution function
explanatory.variable <- seq(from=0, to=100, by=0.1)
response.variable <- dnorm(explanatory.variable, mean=40, sd=2)

# generate plot
plot(explanatory.variable, response.variable, 
     type="l", main="Normal distribution", 
     lty=2)

Readability

Because programming languages are fundamentally a form of simplified English, write for humans, not for computers.

If you have a variable called “Percentage of W-E alignments”, don’t call it “pwe” in your code.

pwe <- c(0.45, 0.5, 0.65)               # bad practice

per.WE.align <- c(0.45, 0.5, 0.65)          # better

percentage.west.east.alignments <- c(0.45, 0.5, 0.65)   # great

Define functions that explain what they do.

avg <- mean.rate(pwe)                               # bad practice

average.rate <- average.rate.from.alignments(percentage.west.east.alignments)   # great

Style guides

Some programming language have official styles. E.g. in Python, it is good style to write as:

name = "Hari"           # variables start with lowercase
last_name = "Seldon"        # variables use underscores to separate words

YEARS_OF_DARKNESS = 1000    # constants are written all caps with underscores

# specific rules about trailing spaces
foundation(terminus[1], {trantor: 2})  # Correct
foundation( terminus[ 1 ], { trantor : 2 } ) # Wrong

Not all languages have an official Style Guide! For the longest time R did not have one, now there is the Google R’s Style Guide but many have developed their own style, e.g.:

last.name <- "Seldon"   # Python would complain with this!

Use affordance to make your code more readable!

“Affordances are the characteristics or properties of an object that suggest how it can be used.” (from here).

Programming to make code modifiable

Making the code modifiable means many things. Mostly, it boils down to modularity and usage of patterns.

Modularity means, for example, to write independent units of code for different tasks, opposed to one long script without breaks. Use functions for tasks you perform regularly:

plot.data.with.regression <- function(explanatory, response, line.color){
  min.y.plot <- min(response) - 1
  max.y.plot <- max(response) + 1
  plot(explanatory, response, 
       xlab="Variable 1", ylab="Response", 
       ylim=c(min.y.plot, max.y.plot),
       main="Data with model",
       pch=5)
  model <- lm(response ~ explanatory)
  lines(explanatory, predict(model), col=line.color, lty=4, lwd = 3)
}

noise <- rnorm(101, 0, 5)
variable.1 <- seq(from=0, to=10, 0.1)
fake.data <- 0.56*expl + noise + 0.5

plot.data.with.regression(variable.1, fake.data, "blue")

Inputs and outputs

Be careful and document what the different pieces of software need as input, and what they produce as output.

Think of your code as a Lego: each action is performed by different pieces, and the pieces have to fit together.

The “teeth” on top of the Lego brick are the inputs, and the holes in the back of the brick are your outputs. They need to fit.

Keeping track of your Lego pieces, and ensure that their inputs/outputs follow a pattern will make modifying your code easier!

Documenting code

Documenting code means describing what a snippet of code does, what input it takes and what output it produces. All published packages come with documentation, e.g. dpylr.

Here is an example of documentation in R:

#' Plot a set of data with a linear regression.
#'
#' @param explanatory A vector of data for the explanatory variable.
#' @param response A vector of data for the response variable.
#' @param line.color A string determining the plot color of the regression.
#' @returns Produces a plot including the explanatory VS the response variable, 
#'          with the regression overlaid.
plot_data_with_regression <- function(explanatory, response, line.color){
  min.y.plot <- min(response) - 1
  max.y.plot <- max(response) + 1
  plot(explanatory, response, 
       xlab="Variable 1", ylab="Response", 
       ylim=c(min.y.plot, max.y.plot),
       main="Data with model",
       pch=5)
  model <- lm(response ~ explanatory)
  lines(explanatory, predict(model), col=line.color, lty=4, lwd = 3)
}

This style of documentation can be used by the package roxygen2 to generate documentation for a package.

Software design

A whole course of software design will last a whole semester, and mastering software design will last the rest of your lives. There are different design philosophies. I mainly work with two.

  • Top-down design
  • “Exploratory” design

Top-down design

Before writing any code, I define broad things that my code is supposed to do. E.g.

# this is pseudo-code!
wakeUpProcedure = function(alarmSound) {
    turnOffAlarm();
    openEyes();
    standUp();
    walkToToilet();
    ...
}

For each component here, I will need to define other functions:

standUp = function() {
    sitInBed();
    turnTowardsBedSide(leftOrRight);
    putFeetOnTheGround();
    ...
}

Programming to make code testable

Testing code = making sure that your code works

Testing includes:

  • printing data to the console to check it
  • using dedicated testing functions (unit testing)
  • benchmarking
  • changing things at random to see if it keeps working
  • debugging

But it is a super broad topic, I highly suggest you read up on it by yourself if you’re interested!

What to look for?

  • Is the code easy to obtain?
  • Does the repository (Github, Zenodo,…) explain what the code does, or do you need to read the paper in detail?
  • How clear is the README?
  • How are the folders structured? Is it easy to understand which file does what?
  • Are there tests? Was benchmarking done in any way?
  • Is there documentation?
  • Ultimately: is this an acceptable software?
